Errors:
    - Unlike other languages where error handling is done via try-catch-finally blocks ,
      in Go it is done differently.
    - Instead ,error handling is done by returning a value of type error as last return value for a function.
    - This started as convention but become so important that it is a rule.
    - It is the job of calling function to incorporate the checking for presence of absence of error into logic.
    - In most cases when error is being returned , other return values should be set to their zero values.
    - Error is a built-in interface which defines a single method : Error()string.
    - Anything implementing this interface is considered an error , and reason for returning nil in absence of error
      is because error is an interface and zero value for error is nil in go.
    - Two ways to create errors in Go : errors.New() and fmt.Errorf() functions.

    - Sentinel Error:
        - Errors which are meant to signal that processing cannot continue because of problem with current state.
        - These errors can only be declared at package level only , and by convention their names start with Err ( only exception being io.EOF).
        - Always better to reuse built in sentinel errors , and only when absolutely required because once creates , these error variables need.
          to maintained for future project scope and backward compatability as well.
        - Presence of sentinel error is tested using == or != , eg: if err==zip.ErrFormat {}.

    - Custom Error:
        - Custom errors can be built upon Error package as stated above , somethings to take care of:
            - never define a variable of type of custom error , either explicitly return nil or define variable to be type of error (built-in) , when no error occurs.

    - Wrapping Error:
        - When error is passed back through your code , often additional information needs to be added to it.
        - The idea of preserving error while adding additional info to it , is called error wrapping.
        - Error Tree: series of wrapped errors is called error tree.
        - Error wrapping is done using fmt.ErrorF and %w special verb.
        - To deal with the error tree / wrapped errors , there are 3 ways:
            - errors.Unwrap(err) : returns the next error in the chain (one level down)
            - errors.Is(err,sentinelError) : checks whether any error in chain matches a target error.
            - errors.As(err,sentinelError) : checks if an error in the chain matches a specific target error and if yes then extract that and assign to a variable.
            - errors.Is and errors.As functions can be implemented for custom error types , which will be used to replace the built in versions of same.
        - In order to wrap an error with custom error type , the custom error type needs to implement the Unwrap method : func Unwrap(err error) error
        - In case new error needs to be created containing message from another error , use fmt.Errorf() , but use %v verb instead of %w.

        - Wrapping Multiple Errors:
            - for tasks which return []error , the errors need to be merged to obey the fact that standard function signatures return error and not []error.
            - errors.Join(errs...): this function allows merging errors together and sending single error obeying function signature at same time.



Panic and Recover:
    - Panic: it is similar to what an error is in java / python. It is a state generated by go runtime whenever it is unable to figure out what should happen next,
      almost everytime due to programming error.
    - When panic hits:
        - current function halts execution.
        - any defers attached to this function runs in lifo format.
        - then defers attached to calling functions run , until main is reached.
        - program then exits (with status code 2) and prints the stack trace.
        - special case: when goroutines panic , then stop at the calling function which called this goroutine at the top of hierarchy , not necessarily main function.

    -Recover: A way to capture panic and provide graceful shutdown or prevent shutdown at all. Built-in recover function (with defer) helps in doing so.
        - panic thrown by runtime or user (using built in panic() method) , both can be caught by recover.
        - recover is always run in a pattern :
            - registered with defer to handle potential panic
            - called within an if function and check if non nil value is found.
        - Recovery from panic should be rare , because when recovery happens you do not get the stack trace and very limited error context is with you.

    General Advice:
        - panic and recover look a lot like exception handling in other languages , but unlike them , panic and recover are meant to be used rarely , only
          for conditions from where resuming program does not make sense like out of memory errors . For all other purpose use error as values concept.



Errors V/S Panics: when to use what?
    - Throw errors for expected failures , like file not found , db connection failure , such issues will be catered by the calling function.
    - Throw panics for unrecoverable scenarios or programmer bugs (index out of bound etc.).
    - First of all panics should be rare in go , and if they are present then they are to  be recovered by using recover function.
